# Euripus Sports API v1 Plan

## 1. Goal

Build a **Sweden-first sports discovery + watch guidance API** for Euripus.

v1 should answer:

- What sports events are live now?
- What starts later today / this week?
- Which providers likely carry the event?
- Which channel or service label should the user search for in Euripus?
- If a PPV / event feed is likely, prefer that over a generic sports channel.

v1 **does not** need to solve exact `Event -> Play now` mapping inside Euripus.
Instead, it should return:

- canonical event metadata
- prioritized watch availability by market
- channel / provider labels when known
- search hints that Euripus or the user can use manually

## 3. Research summary: providers to include

The provider model should distinguish between:

- `provider_family`: brand family the user thinks in
- `channel_name`: concrete channel / feed label if known
- `watch_type`: `ppv-event`, `streaming`, `linear-tv`, `streaming+linear`

Do **not** hardcode too aggressively where rights are unclear or changing. Keep dated rules with confidence scores.

## 5. Event information sources to include

Use a **source reliability tier**.

### Tier A: official competition schedule/results pages
Best source for canonical event truth.

### Tier B: official provider / rights pages
Best source for provider mapping, packages, and branding.

### Tier C: TV guides / editorial guide sites
Best for day-of exact watch labels, channels and timing.

### Tier D: fallback aggregators
Only used if official pages are broken or too incomplete.

```json
    "recommended_provider": "TV4 Play",
    "availabilities": []
  },

```

```json
{
  "market": "se",

```

## 7.2 Endpoints

### `GET /v1/events/live`
- Return live events with ranked watch availabilities.

### `GET /v1/events/upcoming?hours=72`
- Return upcoming events in next window.

### `GET /v1/events/today`
- Best home-screen feed.

### `GET /v1/events/{id}`
- Return full event detail with all watch options and search hints.

### `GET /v1/competitions/{slug}`
- Useful for league screens.

### `GET /v1/providers`
- Optional helper endpoint exposing provider catalog and aliases.

## 8. Recommended Rust implementation

## 8.1 Crates

Suggested stack:

- `axum` - HTTP API
- `tokio` - async runtime
- `tower-http` - middleware (trace, CORS, compression, timeouts)
- `reqwest` - HTTP client
- `scraper` - HTML parsing
- `serde`, `serde_json`, `serde_yaml` - serialization and config
- `time` - time handling
- `sqlx` with SQLite - storage
- `regex` - title parsing
- `strsim` - fuzzy matching for watch inference
- `tracing`, `tracing-subscriber` - logging and instrumentation
- `thiserror` / `anyhow` - errors
- `clap` - CLI/admin jobs
- `utoipa` - OpenAPI generation

Optional later:
- SQLite FTS or `tantivy` for search

**Do not start with a browser automation dependency.**
For v1, prefer static HTML + officially rendered pages. Add headless browsing only if one critical source becomes impossible without it.

## 11. Availability inference engine

This is the heart of v1.

## 11.1 Inference stages

### Stage 1: direct exact provider event match
Example:
- TV4 Play has `Degerfors IF - IF Elfsborg`
- Allsvenskan event has same title and close time
- produce high-confidence watch availability

### Stage 2: competition-level provider rule
Example:
- event = SHL
- no exact same-day provider page hit
- emit `TV4 Play` with medium-high confidence

### Stage 3: guide-site confirmation
Example:
- TVsport or Matchtider says event is on a specific channel
- enrich availability with `channel_name`
- raise confidence

### Stage 4: manual override
Example:
- special cup match
- unusual rights split
- temporary PPV feed naming quirk

---

## 11.2 Ranking rules

### Market order
1. `se`
2. `us`
3. `uk`

### Watch type order
1. `ppv-event`
2. `streaming`
3. `streaming+linear`
4. `linear-tv`
5. `reference`

### Confidence boosts
- exact title match: `+20`
- team names exact + start time within 30m: `+15`
- official provider page: `+15`
- official competition/provider rights page: `+10`
- Swedish guide corroboration: `+10`
- manual override: `+30`

### Confidence penalties
- only competition-level rule, no per-event corroboration: `-10`
- fuzzy title only: `-15`
- stale source (`>24h` for near-term events): `-20`

---

## 11.3 Search hint generation

Generate hints per event + provider.

### Base queries
- `"{home} {away}"`
- `"{competition} {home} {away}"`
- `"{title} {provider}"`

### Golf queries
- `"{tournament} round {n} {provider}"`
- `"{tournament} featured group {provider}"`
- `"{tournament} main feed {provider}"`

### Channel queries when available
- `"{channel_name} {home} {away}"`
- `"{channel_name} {tournament}"`

These should be returned directly in the API.

## 12. PPV / event-feed handling in v1

Even though Euripus will not map directly to a playable stream yet, v1 should still carry PPV-aware semantics.

### Recommendation
If a source uses an event-specific label like:
- `Arsenal - Sporting`
- `RBC Heritage Round 3`
- `Djurgårdens IF - Malmö FF`

then classify it as:
- `watch_type = ppv-event` if it looks like a dedicated event entry
- otherwise `streaming` or `linear-tv`

### Important
Do not try to perfectly detect provider-internal PPV channel IDs yet.
It is enough in v1 to say:
- provider family
- likely event-specific label
- search hints

## 17. Operational notes

- Cache aggressively; these pages do not need second-level freshness.
- Use polite scraping intervals.
- Log source changes and failures.
- Keep source HTML snippets for debugging failed parsers.
- Store raw source payloads short-term for traceability.
- Add `valid_from` / `valid_to` to provider rules because rights change.

## 18. Known risks

### 1. Rights drift
Rights change season to season.
**Mitigation:** rules are dated and overrideable.

### 2. JS / cookie-heavy pages
Some official pages are noisy.
**Mitigation:** prefer officially rendered pages already proven to work; add fallback sources.

### 3. Provider branding fragmentation
Same event may be labeled by service brand, channel brand or feed brand.
**Mitigation:** provider families + alias tables.

### 4. Golf feed complexity
Golf has multiple feeds and windows.
**Mitigation:** model `tournament + round/session` instead of a single monolithic event.

### 5. No exact Euripus playback mapping in v1
**Mitigation:** return strong search hints and ranked provider labels.

## 19. Recommended v1 success criteria

v1 is successful if:

- all target competitions ingest correctly
- Sweden-first provider recommendations are mostly right
- users can reliably find the event manually in Euripus
- search hints are good enough for future automated search-rule integration
- the system is easy to update when rights or source markup changes

---

## 20. Final recommendation

Build v1 around **three layers**:

1. **Canonical event truth** from official competition pages
2. **Provider/watch truth** from official provider pages + Swedish guide sites
3. **Inference + search hints** instead of exact playback mapping

That gives you a practical, maintainable v1 that is already useful inside Euripus, while leaving a clean path to later `Watch now` integration.
